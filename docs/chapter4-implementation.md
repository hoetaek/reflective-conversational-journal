# Ⅳ. 시스템 구현

본 장에서는 3장에서 제시한 논리적 설계가 실제로 구현 가능함을 증명한다. 구체적으로, 각 컴포넌트가 어떻게 구현되었는지, 왜 특정 기술을 선택했는지, 그리고 시스템이 실제로 작동함을 어떻게 검증했는지를 제시한다.

## 1. 구현 환경 및 기술 선택

### 가. 구현 요구사항

3장에서 제시한 4-layer 아키텍처와 설계 원칙을 실제로 구현하기 위해서는 다음 요구사항을 만족하는 기술 환경이 필요하다:

1. **AI 에이전트 기본 요건**
   - 시스템 프롬프트 설정: Persona를 일관되게 유지
   - 자동 맥락 주입: 사용자 프로필 및 과거 저널 자동 로드
   - 워크플로우 확장성: 다양한 저널링 절차를 독립적 모듈로 실행
   - 도구 사용 능력: 파일 읽기/쓰기 등 실질적 작업 수행

2. **템플릿 및 저장 방식**
   - 구조화된 템플릿 지원 (Free Notes → Reflections → Insights)
   - 텍스트 기반 저장 (가독성, 이식성 확보)
   - 로컬 우선 저장 (데이터 주권 보장)

3. **사용자 접근성**
   - 크로스 플랫폼 지원 (웹, 모바일)
   - 일상적 사용 가능한 접근성

### 나. AI 에이전트 플랫폼 선정

본 연구는 설정 중심으로 행동을 제어할 수 있는 사전 구축형 코딩 에이전트 플랫폼을 대상으로 평가를 진행하였다. 이는 설계·개발 연구의 특성상 반복적 설계 검증이 요구되므로, 구현·수정 비용을 낮추기 위함이다.

평가 대상 플랫폼은 세 가지였으며, 시스템 프롬프트 준수도, 자동 맥락 주입, 워크플로우 확장성을 기준으로 비교하였다:

**<표 Ⅳ-1> AI 에이전트 플랫폼 기능 비교**

| 평가 기준 | 플랫폼 A | 플랫폼 B | 플랫폼 C |
|----------|---------|---------|---------|
| 시스템 프롬프트 일관성 | ✓ | ✗ | ✗ |
| 자동 맥락 주입 메커니즘 | ✓ (hooks) | ✗ | ✗ |
| 워크플로우 확장 지원 | ✓ (commands) | 제약 있음* | 불안정** |

\* 설정 파일이 프로젝트가 아닌 홈 디렉토리에 저장되어 공유 제약
\*\* 지시 이탈로 인한 일관성 부족

> **주:** 평가 시점 2025년 기준, 실제 제품명은 각각 Claude Code, Codex CLI, Gemini CLI

평가 결과, 세 가지 요구사항을 모두 충족하는 플랫폼 A를 AI 에이전트 플랫폼으로 선정하였다. 이 플랫폼은 시스템 프롬프트를 통한 Persona 정의, Hook 기능을 통한 자동 맥락 주입, 그리고 프로젝트별 Slash Command 배포 기능을 안정적으로 지원함을 확인하였다.

### 다. 저널 저장 방식

저널 데이터는 마크다운(Markdown) 형식으로 로컬 파일 시스템에 저장된다. 마크다운은 범용적인 텍스트 기반 형식으로 다음과 같은 장점을 제공한다:

- **가독성**: 일반 텍스트 편집기에서 읽기 가능
- **이식성**: 특정 도구에 종속되지 않음
- **구조화**: 템플릿을 통한 일관된 구조 유지
- **프라이버시**: 로컬 저장으로 데이터 주권 보장

## 2. 아키텍처 구현 매핑

3장에서 제시한 논리적 아키텍처의 각 컴포넌트는 다음과 같이 구체적으로 구현되었다:

**<표 Ⅳ-2> 논리적 아키텍처 → 구현 매핑**

| 논리적 컴포넌트 (3장) | 구현 방식 (4장) | 설명 |
|---------------------|----------------|------|
| **Persona** | System Prompt 파일 | 대화 철학(Rogers, Socrates, Dweck)을 시스템 프롬프트로 정의. 모든 대화에서 일관된 페르소나 유지 |
| **Workflow** | Slash Commands | 일일 저널, 주간 회고 등 저널링 절차를 독립적 명령어로 구현. 프로젝트별 커스터마이징 가능 |
| **Context** | Session Hooks | 세션 시작 시 사용자 프로필과 최근 저널을 자동으로 로드하여 맥락 제공 |
| **Reflection** | File Templates + Write Tool | 성찰 프레임워크(What-So What-Now What)를 템플릿으로 정의하고, 대화 내용을 구조화하여 저장 |
| **Storage** | Markdown Files | 로컬 파일 시스템에 `.md` 형식으로 저장. 버전 관리 시스템(Git) 활용 가능 |

### 가. Persona 구현

시스템 프롬프트 파일(`CLAUDE.md`)을 통해 AI의 대화 철학을 정의한다. 본 연구에서는 Rogers의 무조건적 긍정적 존중, Socrates의 탐구적 질문법, Dweck의 성장 마인드셋을 결합한 페르소나를 구현하였다. 이 프롬프트는 모든 대화 세션에서 자동으로 적용되어 일관된 대화 경험을 제공한다.

**교체 가능성:** 다른 대화 철학(예: 코칭형, 멘토링형)으로 페르소나를 변경할 수 있다.

### 나. Workflow 구현

저널링 절차를 Slash Command로 구현한다. 예를 들어 `/journal` 명령어는 일일 저널링 워크플로우를, `/weekly` 명령어는 주간 회고 워크플로우를 실행한다. 각 워크플로우는 독립적인 설정 파일로 관리되어 프로젝트별로 커스터마이징 가능하다.

**교체 가능성:** 새로운 저널링 절차를 추가하거나 기존 절차를 수정할 수 있다.

### 다. Context 구현

Session Hook을 통해 대화 시작 시점에 자동으로 맥락을 로드한다. Hook은 다음 정보를 읽어들여 AI에게 제공한다:

- 사용자 프로필 (`profile.md`): 이름, 가치관, 목표 등 장기 기억
- 최근 저널 (예: 최근 3일): 단기 맥락

**교체 가능성:** 맥락 로딩 전략을 변경할 수 있다 (예: 최근 N일, 주간 요약본).

### 라. Reflection 구현

성찰 프레임워크를 마크다운 템플릿으로 정의한다. 본 연구에서는 Borton(1970)의 What-So What-Now What 프레임워크를 채택하였으며, 템플릿 구조는 다음과 같다:

```markdown
# [날짜] 일일 저널

## Free Notes
- [시간] 사용자 발화
- [시간] AI 응답
...

## Reflections
### What (무엇을 경험했는가)
- ...

### So What (그것이 의미하는 바는)
- ...

### Now What (다음에 어떻게 할 것인가)
- ...

## Long-term Insights
- ...
```

AI 에이전트는 대화 내용을 분석하여 이 템플릿에 맞춰 자동으로 구조화한다.

**교체 가능성:** Gibbs Reflective Cycle, Kolb Learning Cycle 등 다른 성찰 프레임워크로 교체 가능하다.

### 마. Storage 구현

모든 저널은 로컬 파일 시스템의 `journals/` 디렉토리에 마크다운 파일로 저장된다. 파일명은 날짜 기반으로 자동 생성되며(예: `2025-12-18.md`), Git을 통한 버전 관리도 가능하다.

**장점:**
- 데이터 주권: 모든 데이터가 로컬에 저장
- 이식성: 다른 마크다운 도구에서도 읽기 가능
- 영속성: 플랫폼 변경에도 데이터 유지

## 3. 구현 결과

### 가. 시스템 구성

최종 구현된 시스템은 다음과 같이 구성된다:

```
├── journals/
│   ├── profile.md              # 사용자 프로필
│   └── 2025-12-18.md          # 일일 저널 예시
└── templates/
    └── daily-journal.md        # 저널 템플릿
```

### 라. Pluggability 검증

3장에서 강조한 Pluggable Architecture가 실제로 구현 가능함을 검증하기 위해, Python을 사용한 프로토타입을 구현하였다. 이 프로토타입은 다음을 증명한다:

#### 1) 각 컴포넌트의 교체 가능성

```python
# Persona 교체
persona = Persona(definition=RogersSocratesDweck)
# → 다른 Persona로 교체: PersonaDefinition("코칭형", {...})

# Workflow 교체
workflow = Workflow(definition=DailyJournal)
# → 다른 Workflow로 교체: WorkflowDefinition("주간 회고", [...])

# Context 전략 교체
context = Context(strategy=RecentDays(days=3))
# → 다른 전략으로 교체: WeeklySummaryStrategy()

# Reflection 프레임워크 교체
reflection = Reflection(framework=WhatSoWhatNowWhat)
# → 다른 프레임워크로 교체: GibbsReflectiveCycle()
```

#### 2) Strategy 패턴 검증

모든 컴포넌트가 Strategy 패턴을 통해 독립적으로 교체 가능함을 확인하였다. 각 전략은 명확한 인터페이스를 구현하며, 시스템의 다른 부분에 영향을 주지 않고 교체될 수 있다.

#### 3) 프로토타입 위치

- Repository: `architecture_prototype.py`
- 전체 4-layer 아키텍처 구현
- Dependency Injection 패턴 적용
- 실행 가능한 대화형 시스템

### 마. 일상적 사용 검증

본 연구자는 2024년 X월부터 현재까지 이 시스템을 일상적으로 사용하여 다음을 확인하였다:

- **일관성**: Persona가 모든 대화에서 일관되게 유지됨
- **맥락 연결성**: 과거 저널 내용이 대화에 자연스럽게 반영됨
- **구조화**: 자유로운 대화가 체계적인 성찰 구조로 변환됨
- **확장성**: 새로운 워크플로우 추가가 용이함
- **Pluggability**: 컴포넌트 교체가 실제로 작동함

이를 통해 3장에서 제시한 논리적 아키텍처가 실제로 구현 가능하며, 일상적 사용에도 적합함을 실증하였다.
